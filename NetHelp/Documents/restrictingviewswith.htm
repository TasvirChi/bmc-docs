Restricting Views with a Borhan Session

Often, the user authentication/entitlement mechanism that decides whether a user is entitled to access a specific media entry or not, will reside outside of Borhan. When the authentication mechanism resides outside Borhan, we recommend that you use an Advanced Borhan Session (KS) based Access Control mechanism, that permits access to the media only when a valid Borhan Session is provided. .The external entitlement mechanism will then generate the valid Borhan Session when users should be permitted to view the content.

When implementing paid content where the payment gateway is not Borhan, the server processing the payment will be the mechanism deciding on entitlement. After the payment is processed, the payment server creates a valid Borhan Session by calling the Borhan API and passes that KS to the Borhan player.

When implementing a single-sign-on and permissions mechanism using LDAP or other authentication mechanism outside of Borhan, the server responsible for user authentication contains the entitlement logic. The authentication server will call the Borhan API and generate a valid KS when appropriate.

A Borhan Player marked with tokenized access requires a “session token” to be created and provided during every playback. A session token can only be created by a developer possessing valid credentials and expires after a limited amount of time. Thus, even if an attempt to grab the item’s direct URL is successful, the playback session will soon expire and without possession of the valid credentials, regeneration of a new token will not be possible.

This option is useful to prevent scraping of content from your website or re-sharing of content on other sites that are not yours and provides an additional amount of security to content display.
